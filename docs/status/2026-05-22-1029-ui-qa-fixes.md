## Important Discoveries — root-cause analysis per issue

### Issue 7 (DONE) — chat bobs on keystroke

Root cause: `ChatShell`'s scroll effect depends on `[children]`, which is a
fresh array on every parent render — i.e. **every Composer keystroke**. It
called `scrollTo({behavior:"smooth"})` each time; the smooth animation never
settled → perpetual 5-10px bob. Only visible at 2+ composer lines because
that's when `<main>` content actually overflows. Fix applied: only scroll when
`<main>.scrollHeight` changed. A second candidate — the Composer autosize
`useEffect` with the `height="0px"` reset — was analysed and **ruled out**:
within a stable-line-count textarea the two synchronous style writes never
paint between, so it cannot bob per-keystroke. Did NOT change the autosize
effect (KARPATHY: touch only what you must).

### Issue 4 (TODO) — freetext lost on questionnaire back-nav

File: `src/components/chat/Composer.tsx`. Root cause: `value` is a single
parent-owned string (no per-step storage). On freetext send, `handleSend`
clears it (`onChange("")`) and commits the answer into `answers[step]`. The
textarea is `disabled={stepLocked}` where `stepLocked = answers[step] != null`
— so a freetext step locks exactly like an MC step, and its text is gone on
revisit. **Proposed approach** (designed, not yet built): add a per-step
`drafts: string[]` state; in questionnaire mode the textarea reads/writes
`drafts[step]` instead of `value`; free-text steps never lock the textarea
(`currentIsFreeText = current.options.length === 0`; `textareaLocked =
stepLocked && !currentIsFreeText`); `handleSend` no longer clears the draft;
`onComplete` reads `drafts[i]` for free-text questions (picks up back-edits)
and `answers[i]` for MC. Reset effect must also `setDrafts(Array(len).fill(""))`.
MC keeps current lock-on-confirm behaviour. Composer.tsx is a "parity-locked
port" with file-wide `eslint-disable` for `max-lines` + `functional/immutable-data`,
so adding state is within tolerance.

### Issue 5 (TODO) — questionnaire lingers in composer after submit

Files: `Onboarding.tsx`, `WaveSession.tsx`. Root cause: the Composer's
`questions` prop comes from `activeQuestionnaire`, derived from server state;
it stays non-null until the server round-trip refetches. **Proposed:** parent
tracks a `submittedKey` (the `questionsKey`/questionnaireId just submitted in
`onComplete`); pass `questions={null}` while `activeQuestionnaire.questionsKey
=== submittedKey`. Naturally clears when server state advances to a different
questionnaire or none.

### Issue 6 (TODO) — framework double-trip duplicate bubble

**Confirmed: framework creation IS two LLM calls.** `useScopingState`
(`src/hooks/useScopingState.ts` lines ~117-139) has an effect that
auto-dispatches `generateBaseline` once `framework` lands and `baseline` is
null — no user action between. → **GH issue** to merge them into one call
(out of scope, user-requested). In-scope bug: after framework lands,
`deriveTurns` emits the real clarify-answers turn, but `isPending` is still
true (baseline auto-dispatching) and `pendingMessage` was never cleared → the
optimistic bubble renders a 2nd time. **Proposed:** a pure derived guard —
don't render the optimistic bubble if a turn already exists in `turns` with
matching `content`. `formatComposerAnswers` is byte-identical to deriveTurns'
`formatAnswers` (its TSDoc says so), so an exact string match is reliable.

### Issue 8 (TODO) — error handling

Files: `useScopingState.ts`, `useWaveState.ts`, `TopicInput.tsx`. Root causes:
(a) the 3 scoping mutations (`generateFramework`, `generateBaseline`,
`submitBaseline`) have **no `onError`** → failures are silent, no toast.
(b) the optimistic bubble is gated on `isPending` only → it vanishes the
instant `isPending` clears on error. **Proposed:** a pure helper
`formatMutationError(err)` (TDD it, e.g. `src/lib/errors.ts` + colocated test)
that extracts `err.data.httpStatus` (→ "HTTP 429") or `err.data.code`
(→ "TOO_MANY_REQUESTS") and appends it to the toast description; add `onError`
toasts to the 3 mutations (friendly title, code in description); expose a
`submitError` boolean from `useScopingState`/`useWaveState`; render the
optimistic bubble while `(isPending || submitError)`. `useWaveState` already
has an `onError` on `submitTurn` — just enhance its description with the code.
`TopicInput`'s `clarify` onError already drops the optimistic topic
intentionally ("so the learner can retry" via EmptyState) — leave that, only
add the code to its toast. **User said the actual retry/recovery is backend
and out of scope** — in-scope is purely the frontend error surfacing.
